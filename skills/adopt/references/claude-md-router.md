# Роутинговый слой для проектного CLAUDE.md («принцип чемодана»)

Используется шагом 3.8 скилла `/adopt` для проектов, чей `CLAUDE.md` перерос
роль entry-файла. Принцип: **часто нужное — под рукой, иногда нужное — убрать
поглубже, никогда не нужное — выкинуть**. Entry-файл — короткий и роутинговый;
темы — в отдельных доках, которые агент читает только при необходимости.

## Целевая структура

```
CLAUDE.md              ← entry-роутер, 50–200 строк (жёсткий потолок ~250)
docs/claude/*.md       ← тематические доки, 50–150 строк каждый
```

Entry-роутер содержит ТОЛЬКО:

1. **Обзор проекта** — 1–2 предложения (что это, стек, где прод).
2. **Команды первого запуска** — установка / тест / полная проверка (3–6 строк).
3. **Жёсткие ограничения** — не больше **15** необсуждаемых правил. Правило
   попадает сюда, только если его нарушение = инцидент (деплой, деньги,
   безопасность, потеря данных). Всё остальное — в тематический док.
4. **Индекс тематических доков** — по одной строке на тему:
   `- <Тема> (docs/claude/<file>.md) — <когда читать: условие применимости>`
5. Блок методологии `<!-- idea-to-deploy:begin -->…end` (если проект адоптирован).

Маркер идемпотентности: `<!-- itd:claude-router -->` в первой сотне строк
entry-файла — повторный прогон шага 3.8 при его наличии ничего не предлагает.

## Правила нарезки тем

- **Одна тема = один док**: деплой/инфра; доменные правила подсистемы; правила
  БД/миграций; интеграции (по одной на внешнюю систему, если крупные); тестовые
  конвенции; исторические инциденты и их уроки. Как органайзеры в чемодане —
  не нужно вытряхивать всё, чтобы найти зарядку.
- **Строка индекса несёт условие применимости**, не только название:
  «— читать при правке проведения документов», «— читать перед деплоем».
  Агент решает по строке, открывать ли док; расплывчатое «полезная инфа» =
  док никогда не откроют или будут открывать всегда.
- **Информация в коде остаётся в коде**: определения типов, комментарии к
  интерфейсам, пояснения в конфигах НЕ дублируются в доках — агент увидит их
  при чтении кода. В док идёт только то, что из кода не выводится (почему, а
  не что; договорённости; внешние ограничения).
- **Changelog / история сессий** — не грузятся в контекст вообще: отдельный
  файл (`CLAUDE-changelog.md` и т.п.), упомянутый в индексе одной строкой.
- **Край, не середина**: то немногое, что обязано жить в entry-файле,
  ставится в начало или в конец, никогда в середину («lost in the middle»).
  Но лучший ответ на «важную инструкцию в середине» — вынести её в topic-док.

## Lifecycle перенесённых инструкций

У каждой инструкции в entry и в topic-доках должно быть ясно: **откуда она**
(источник/дата/версия), **когда применяется** и **когда её можно удалить**.
При разрезании — выкидывай устаревшее и противоречивое, не переноси мусор в
новые доки: ревизия — часть сплита, «утрамбовать в органайзеры весь хлам» не
является целью. Управляй инструкциями как зависимостями кода — неиспользуемые
удаляются, иначе только тормозят систему.

## Процедура сплита (шаг 3.8 /adopt)

1. Прочитай текущий `CLAUDE.md`, размети секции по темам (список тем — юзеру).
2. Покажи план: какие секции остаются в entry (и почему), какие уходят в какой
   `docs/claude/*.md`, какие предлагаешь УДАЛИТЬ (устаревшие/дублирующие код).
3. **Только после явного подтверждения** пользователя выполняй перенос:
   контент переносится ДОСЛОВНО (не пересказ — сохранить точность команд,
   путей, чисел); удаления показываются отдельным списком.
4. В entry вставь маркер `<!-- itd:claude-router -->` и индекс тем.
5. Проверь: entry ≤ 200–250 строк; каждый topic-док ≤ ~150 строк (крупные темы
   дели на под-темы); суммарный контент не потерян (`wc -l` до/после ± удалённое).

## Шаблон entry-роутера

```markdown
<!-- itd:claude-router -->
# <Проект>

<1–2 предложения: что это и на чём.>

## Быстрый старт
- Установка: `<cmd>`
- Тесты: `<cmd>`
- Полная проверка: `<cmd>`

## Жёсткие ограничения (нарушение = инцидент)
1. <правило>
… (≤ 15)

## Тематические доки (читать по условию)
- Деплой и сервер (docs/claude/deploy.md) — перед любым деплоем/правкой инфры
- <Подсистема X> (docs/claude/x.md) — при правке <области>
- Инциденты и уроки (docs/claude/incidents.md) — при диагностике «уже видели такое»
- Changelog (CLAUDE-changelog.md) — не грузить; смотреть точечно по датам

<блок idea-to-deploy, если есть>
```
