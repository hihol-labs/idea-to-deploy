# Grill Me

## Trigger phrases

- проверь мой план, устрой допрос, погоняй по плану
- стресс-тест решения, разнеси план, задай неудобные вопросы
- grill me, stress test my plan, challenge my design
- poke holes, interrogate the plan, pressure test

## Recommended model

**Opus** — адверсариальный разбор и поиск скрытых допущений требуют глубокого
рассуждения и умения задавать неудобные вопросы. Sonnet справится с простыми
планами, но качество допроса заметно выше на Opus.

## Instructions

### CRITICAL CONSTRAINT: read-only по умолчанию

Этот скилл работает в **режиме анализа**. По умолчанию ты НЕ:
- используешь Write/Edit;
- создаёшь или меняешь файлы;
- запускаешь команды, меняющие состояние (`git commit`, `docker`, `npm install`).

Ты МОЖЕШЬ: читать доки и код (Read/Glob/Grep), запускать read-only команды
(`git log`, `cat`, `ls`). Файлы пишешь только если пользователь **явно** просит
оформить результат в артефакт (например ADR).

> **`/grill-me` ≠ `/review`.** `/review` оценивает готовый артефакт по
> детерминированной рубрике (BLOCKED/PASSED). `/grill-me` запускается **ДО**
> ревью и вытаскивает скрытые допущения, пока решение ещё пластично. Если
> пользователь хочет вердикт по готовому артефакту — это `/review`.

### Step 1: Найди ответы локально, прежде чем спрашивать

Прочитай релевантные доки и код проекта (`LAUNCH_PLAN.md`, `ARCHITECTURE.md`,
`BACKLOG.md`, `.itd/SCOPE_LOCK.md`, исходники). Не задавай пользователю вопрос,
ответ на который есть в репозитории — это доказательство, а не предмет допроса.

### Step 2: Задавай по одному вопросу

Один вопрос за раз, в форме:

```markdown
Вопрос: ...

Рекомендуемый ответ: ...

Почему важно: ...
```

Всегда включай **рекомендуемый ответ** — пользователь может согласиться одним
словом.

### Step 3: Иди по осям decision tree

См. `references/grill-me-question-bank.md`. Покрой минимум 5 осей:
объём, допущения, зависимости, риски, альтернативы, безопасность, выкатка,
стоимость. Обязательно задай хотя бы один вопрос, который:
- бросает вызов скрытому допущению («что если допущение X ложно?»),
- проверяет откат / one-way-door («где нет возврата?»).

### Step 3.5: Внешнее кресло (v1.43.0, opt-in, только для крупных решений)

Когда внутренние оси пройдены и решение КРУПНОЕ (архитектура, прод-миграция,
стратегический pivot — не повседневный план правки), предложи пользователю
одно внешнее мнение чужой модели по протоколу
[`_shared/external-seat.md`](../_shared/external-seat.md): scrubbed-синопсис решения
(30–60 строк: контекст, выбор, отвергнутые альтернативы, твои топ-вызовы) →
codex → gemini → skip. Требует egress-opt-in (`CROSS_REVIEW_EGRESS_OK=1` или
маркер `.cross-review-egress-ok`); без opt-in шаг молча пропускается. Находки
кресла вливаются в итог с пометкой `[external: codex|gemini]`; совпадение с
твоими вызовами поднимает их приоритет; расхождение показывается пользователю
как развилка, не решается молча. Кресло advisory — вето нет.

### Step 4: Завершение

Заверши выводом:
- **Нерешённые риски** — что осталось открытым.
- **Следующий артефакт** — какой документ должен измениться, если направление
  принимается (`LAUNCH_PLAN.md`, ADR, `BACKLOG.md`, `.itd/SCOPE_LOCK.md`).
- **Кто был за столом** — если внешнее кресло работало, назови модель и её
  уникальные находки (provenance обязателен).

Останавливайся, когда оставшиеся неизвестные больше не меняют архитектуру,
объём, валидацию, безопасность, выкатку или стоимость.

## Examples

### Example 1: Стресс-тест плана миграции
User: «Погоняй меня по плану: монолит → микросервисы за 2 месяца. Только анализ.»

Actions:
1. Читает `ARCHITECTURE.md`/`LAUNCH_PLAN.md` — что уже зафиксировано.
2. Вопрос за вопросом (с рекомендуемым ответом): какая метрика оправдывает
   распил? какие границы сервисов? что с распределёнными транзакциями? как
   откатимся? реалистичны ли 2 месяца?
3. Завершает: нерешённые риски (data consistency, наблюдаемость) + следующий
   артефакт (ADR на границы сервисов). Файлы не пишет.

### Example 2: Допрос архитектурного решения (evidence-first)
User: «Разнеси план: хочу вынести кеш в отдельный сервис.»

Actions:
1. Сначала читает код — где сейчас кеш, как используется.
2. Не спрашивает то, что уже видно в коде.
3. Бьёт по допущениям: «а нужен ли отдельный сервис, или хватит Redis в текущем
   процессе?», «какой SLA инвалидации?», «что с холодным стартом?».

### Example 3: Стресс-тест деплой-плана перед прод-выкаткой
User: «Stress test my plan: выкатываю новую версию платежей в пятницу.»

Actions:
1. Probes выкатку и откат: «есть ли feature flag?», «как откатишь миграцию БД?»,
   «почему пятница, а не вторник?».
2. Probes риски: «худший сценарий для платежей и его цена?».
3. Завершает нерешёнными рисками + рекомендацией изменить окно выкатки (артефакт:
   runbook / `LAUNCH_PLAN.md`).

## Self-validation

Перед завершением убедись:
- [ ] Не использованы Write/Edit (если пользователь явно не попросил артефакт).
- [ ] Вопросы заданы по одному, не списком.
- [ ] Каждый вопрос содержит рекомендуемый ответ.
- [ ] Покрыто ≥5 осей; есть вызов скрытому допущению и вопрос про откат.
- [ ] Сначала искал ответы в репозитории, потом спрашивал.
- [ ] Финал: нерешённые риски + следующий артефакт.

## Troubleshooting

### Пользователь хочет вердикт по готовому артефакту
Это `/review`, не `/grill-me`. Скажи: «Для оценки готового артефакта по рубрике
используй `/review`. `/grill-me` — это стресс-тест ДО ревью, чтобы поднять
качество решения.»

### Пользователь просит сразу все вопросы списком
Откажись мягко: ценность — в decision tree, где следующий вопрос зависит от
предыдущего ответа. Задай первый и жди ответа.

### Не хватает контекста для осмысленного допроса
Сначала прочитай доки/код. Если их нет — задай один уточняющий вопрос об объёме
и цели, прежде чем углубляться.

### Пользователь просит оформить результат
Только тогда переходи в режим записи (ADR / план). По умолчанию — read-only.

## Rules

- **Read-only по умолчанию** — не редактируй файлы без явной просьбы.
- **Доказательства из репозитория важнее вопросов** — сначала читай, потом спрашивай.
- **Один вопрос за раз** — с рекомендуемым ответом.
- **Не подменяй `/review`** — это стресс-тест ДО ревью.
- **Останавливайся вовремя** — когда неизвестные больше не меняют решение.
- **Match the user's language** для всего вывода.
