# Handoff

## Trigger phrases

- хендофф, сделай хендофф, передай контекст
- передача сессии, контекст для передачи, пакет контекста
- handoff, make a handoff, hand off
- transfer context, session handoff, context packet

## Recommended model

**Sonnet** — компактный пакет контекста из уже принятых решений и состояния
репозитория. Это структурированная суммаризация, а не глубокое рассуждение;
Opus избыточен. Haiku допустим для совсем коротких передач.

## Instructions

Передай **actionable** контекст от одной сессии (роли, агента) к следующей,
когда обратного пути нет. `/handoff` — это **передача перед стартом следующего
актора**, а не сохранение вехи для будущего «я».

> **`/handoff` ≠ `/session-save`.** `/session-save` фиксирует контекст по
> завершении вехи (continuity для будущего «я» + `.active-session.lock`).
> `/handoff` пишет компактный пакет для **другого** актора (агент / роль /
> сессия / AFK-прогон), который начнёт работу дальше. Если веха завершена и
> следующего актора нет — это `/session-save`.

### Когда вызывать

- Планирующая сессия передаёт работу реализующей.
- Долгая сессия вот-вот компактится или рестартует.
- Работа делегируется другому агенту, роли или AFK-прогону.
- Восстановление после неоднозначного/проваленного гейта.
- Релиз/PR/CI/внешний инструмент требует ограниченного пакета контекста.

**Порог контекста — 60% (v1.70.0, кодифицировано).** Не жди компакции:
- если по оценке задача потребует **больше 60% контекстного окна** — заложи
  handoff в план с самого старта (пиши durable-артефакты по ходу, а не в конце);
- если израсходовано **~60% окна**, а конец работы в этой сессии не виден —
  пиши `HANDOFF.md` **сейчас**. После авто-компакции пакет писать уже не из
  чего: детали, ради которых он существует, потеряны.
Оценку «сколько окна съедено/потребуется» делай по ходу (объём прочитанных
файлов/логов, длина сессии); точный процент не важен — важно готовить передачу
ДО того, как харнес принудит к ней.

### Step 1: Собери состояние (read-only)

- `git log --oneline -10` + `git status --short` — что закоммичено, есть ли WIP.
- Прочитай актуальные durable-артефакты: `STATE.json`, `GOAL.json` (если
  ведётся долгая цель `/goal`), `BUILD_PLAN`,
  `LAUNCH_PLAN.md`, `BACKLOG.md`, `.itd/SCOPE_LOCK.md`, `.itd/FORBIDDEN_CHANGES.md`,
  `.itd/DECISIONS.md` (журнал решений — поле 4 «Финальные решения» опирается
  на него, а не на пересказ чата; v1.70.0) — всё если есть.
- Если есть `.itd-memory/GOAL.json` — раздел «Состояние» генерирует **репортёр
  харнеса**, не пересказ (v1.45.0): вставь его вывод в `HANDOFF.md` как есть
  (прозу добавляй вокруг, цифры не меняй):

  ```bash
  GT="skills/goal/scripts"; [ -f "$GT/itd_goal_report.py" ] || GT="$HOME/.claude/skills/goal/scripts"
  SHD="skills/_shared"; [ -f "$SHD/itd_py.sh" ] || SHD="$HOME/.claude/skills/_shared"
  sh "$SHD/itd_py.sh" "$GT/itd_goal_report.py" --goal .itd-memory/GOAL.json
  ```
- Зафиксируй текущий узел/юнит и что именно в процессе.

### Step 2: Напиши `HANDOFF.md` (9 обязательных полей)

См. `references/handoff-checklist.md`. Обязательны все 9:

1. **From → To** — из какой роли/сессии в какую.
2. **Причина передачи** — компакция / делегирование / AFK / восстановление / релиз.
3. **Текущее состояние** — узел, юнит, сделано, в процессе.
4. **Финальные решения** — только итоговые (без пересказа обсуждения).
5. **Требуемые входы** — что прочитать/иметь, чтобы продолжить.
6. **Зоны записи и запреты** — что можно менять, что трогать нельзя.
7. **Команды проверки** — как убедиться, что всё работает.
8. **Блокеры и риски** — что мешает; что может пойти не так.
9. **Первое действие** — точный первый шаг (одна строка, `> [!todo]`).

Файл — обычный Markdown, но Obsidian-совместимый: frontmatter
(project/stage/roles), вики-ссылки на артефакты (`[[STATE]]`, `[[BUILD_PLAN]]`),
callouts `> [!todo]` для первого действия и `> [!warning]` для рисков.

### Step 3: Обнови срез состояния

Обнови `STATE.json` в memory-dir проекта (схема-образец:
`docs/templates/itd-memory/STATE.json`) — текущий узел, статус, ссылка на
`HANDOFF.md`. Если ведётся долгая цель (`/goal`) — актуализируй статусы юнитов
в `.itd-memory/GOAL.json` (схема: `docs/templates/itd-memory/goal.schema.json`),
чтобы получатель продолжил с первого не-verified юнита. Если в проекте есть
`.itd/itd_progress.py` — перегенерируй glance-вью (best-effort, при ошибке не
блокируй) → `.itd-memory/PROGRESS.md` (v1.70.0). На Windows `python3` может
быть Store-заглушкой (см. валидацию интерпретатора в `sync-to-active.sh`) —
поэтому с фолбэком (v1.71.1):

```bash
python3 .itd/itd_progress.py 2>/dev/null || python .itd/itd_progress.py 2>/dev/null || py -3 .itd/itd_progress.py 2>/dev/null || true  # win-ok: цепочка включает py -3
```

Если ни один интерпретатор не сработал — шаг тихо пропускается (вью derived,
канон не страдает); при желании задай `ITD_WIN_PYTHON` и вызови им напрямую.
Для публикации пакета в Obsidian-vault — `/obsidian-export`
(если подключён).

### Step 4: Self-validation

Перед выводом проверь по чеклисту в `references/handoff-checklist.md`:
поля 1–9 заполнены, первое действие однозначно, команды проверки указаны,
секретов нет, ссылки на durable-артефакты, а не на историю чата.

## Examples

### Example 1: Передача перед компакцией (planning → implementation)
User: «Сделай хендофф: сессия компактится. Передай контекст по auth.»

Actions:
1. `git log`/`git status` — login/register закоммичены, refresh — WIP (red-тест).
2. Пишет `HANDOFF.md`: From planner → To implementer; причина — компакция;
   состояние (auth, узел refresh-token); финальные решения (python-jose, Redis);
   входы (`[[STATE]]`, `[[BUILD_PLAN]]`); зоны записи (`src/auth/**`); проверка
   (`pytest tests/test_auth.py`); риск (нужен `STRIPE_KEY`); первое действие
   (`> [!todo]` запусти red-тест refresh, затем реализуй эндпоинт).
3. Обновляет `STATE.json`. Секретов в файл не кладёт.

### Example 2: Делегирование агенту на AFK-прогон
User: «Передай контекст агенту, он продолжит ночью.»

Actions:
1. Делает `HANDOFF.md` самодостаточным — без опоры на этот чат.
2. Перечисляет все входы, которые получатель обязан прочитать.
3. Первое действие однозначно — свежий агент может начать «с холода».

### Example 3: Восстановление после проваленного гейта
User: «`/review` упал, передай контекст на восстановление.»

Actions:
1. Фиксирует, какой именно гейт упал и почему (цитата вердикта, не пересказ).
2. В блокерах — что чинить; в первом действии — конкретный фикс-шаг.
3. Зоны записи ограничивает затронутыми файлами.

## Self-validation

Перед выводом убедись:
- [ ] `HANDOFF.md` содержит все 9 полей (или явные `n/a`).
- [ ] Первое действие — одна конкретная команда/шаг в `> [!todo]`.
- [ ] Указаны команды проверки.
- [ ] Указаны зоны записи и запреты.
- [ ] В файле нет секретов/ключей/токенов.
- [ ] Ссылки на durable-артефакты (`[[STATE]]`), а не на историю чата.
- [ ] `STATE.json` обновлён.
- [ ] `GOAL.json` актуален (если ведётся долгая цель `/goal`).

## Troubleshooting

### Пользователь на самом деле хочет сохранить веху
Если следующего актора нет, а работа завершена — это `/session-save`, не
`/handoff`. Скажи: «Похоже на завершение вехи — используй `/session-save`.
`/handoff` нужен для передачи перед стартом следующего актора.»

### Нет durable-артефактов (STATE.json/BUILD_PLAN отсутствуют)
Собери состояние из git-истории и кода, опиши явно в `HANDOFF.md` и пометь
отсутствующие артефакты. Не выдумывай решения, которых не было.

### Слишком длинный пакет
Режь до сути: только финальные решения и первое действие. Детали — через
ссылки на артефакты, а не копипастой в `HANDOFF.md`.

### Риск утечки секретов
Никогда не вставляй ключи/токены/пароли. Вместо значения — имя переменной и
где её взять («нужен `STRIPE_KEY` от заказчика»).

## Rules

- **Не храни секреты** в `HANDOFF.md`.
- **Только финальные решения** — без пересказа обсуждения.
- **Компактно** — свежая сессия читает это первым.
- **Durable-ссылки, не чат** — `[[STATE]]`, а не «как обсуждали выше».
- **Первое действие однозначно** — одна команда/шаг.
- **`/handoff` для передачи, `/session-save` для вехи** — не путай.
- **Match the user's language** для всего вывода.
