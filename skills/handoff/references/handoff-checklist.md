# Handoff — Checklist

Чеклист обязательных полей и критериев качества для `HANDOFF.md` — компактного
пакета контекста, который передаёт работу от одной сессии (роли, агента) к
следующей, когда **обратного пути нет** (компакция, рестарт, делегирование,
AFK-прогон, восстановление после неоднозначного гейта).

> `/handoff` ≠ `/session-save`. `/session-save` фиксирует контекст по завершении
> вехи (для будущего «я»). `/handoff` — передача **перед** тем, как начнёт
> действовать **другой** актор (агент/роль/сессия). Используй `/session-save`
> после вехи, `/handoff` — до старта следующего.

> **Порог контекста — 60% (v1.70.0).** Задача по оценке требует >60% окна →
> закладывай handoff с самого старта; израсходовано ~60% окна и конец не виден →
> пиши `HANDOFF.md` сейчас, не дожидаясь авто-компакции (после неё детали для
> пакета уже потеряны).

## Обязательные поля `HANDOFF.md`

| # | Поле | Описание |
|---|------|----------|
| 1 | **From → To** | Из какой роли/сессии в какую (например `planning → implementation`) |
| 2 | **Причина передачи** | Компакция / делегирование / AFK / восстановление / релиз |
| 3 | **Текущее состояние** | Узел, юнит, что сделано, что в процессе |
| 4 | **Ключевые решения** | Только финальные решения (без истории обсуждения) |
| 5 | **Требуемые входы** | Что нужно прочитать/иметь, чтобы продолжить |
| 6 | **Зоны записи и запреты** | Что можно менять, что трогать нельзя (`.itd/FORBIDDEN_CHANGES.md`) |
| 7 | **Команды проверки** | Как убедиться, что всё работает (тесты, линт, healthcheck) |
| 8 | **Блокеры и риски** | Что мешает; что может пойти не так |
| 9 | **Первое действие** | Точный первый шаг для принимающей сессии (одна строка) |

## Формат файла (Markdown + Obsidian-совместимый)

```markdown
---
project: <path>
stage: implementation
from_role: planner
to_role: implementer
reason: compaction
---

# Handoff — <дата>

> [!todo] Первое действие
> Запусти `pytest tests/test_auth.py -k refresh` — должен быть red, затем реализуй refresh-эндпоинт.

## Состояние
- Узел: auth-flow, юнит: refresh-token
- Сделано: login/register + тесты (12 шт, green)
- В процессе: refresh-эндпоинт (red-тест написан)

## Финальные решения
- python-jose, Redis для сессий (см. [[STATE]])

## Требуемые входы
- [[BUILD_PLAN]], [[STATE]], `.itd/SCOPE_LOCK.md`

## Зоны записи / запреты
- Менять: `src/auth/**`, `tests/test_auth.py`
- НЕ трогать: `.itd/FORBIDDEN_CHANGES.md`, схему БД без миграции

## Проверка
```bash
pytest tests/test_auth.py
ruff check src/auth
```

> [!warning] Блокеры и риски
> Нужен `STRIPE_KEY` от заказчика для следующего юнита (платежи).
```

## Куда писать

- `HANDOFF.md` — в корень проекта (рабочий, перезаписываемый артефакт).
- `STATE.json` — обновить срез состояния в memory-dir
  (`docs/templates/itd-memory/STATE.json` как образец схемы; для проекта —
  его реальный `STATE.json`).

## Obsidian-слой

- Линкуй ключевые артефакты вики-ссылками: `[[STATE]]`, `[[BUILD_PLAN]]`,
  `[[BACKLOG]]`, `[[LAUNCH_PLAN]]`.
- Первое действие — callout `> [!todo]`.
- Риски и блокеры — callout `> [!warning]`.
- Для публикации в vault используй `/obsidian-export` (если подключён).

## Критерии качества

- **Компактно** — свежая сессия должна прочитать это первым и сразу понять, что делать.
- **Только финальные решения** — без пересказа обсуждения.
- **Первое действие однозначно** — одна команда/шаг, не «разберись с auth».
- **Никаких секретов** — не клади ключи/токены в `HANDOFF.md`.
- **Ссылки на артефакты, а не на чат** — `[[STATE]]`, а не «как обсуждали выше».

## Self-validation (перед выводом)

- [ ] Поля 1–9 заполнены или явно помечены `n/a`.
- [ ] Первое действие — одна конкретная команда/шаг.
- [ ] Указаны команды проверки.
- [ ] Указаны зоны записи и запреты.
- [ ] Нет секретов.
- [ ] Ссылки на durable-артефакты, а не на историю чата.
