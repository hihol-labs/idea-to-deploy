# Как я превратил Claude Code из хаотичного генератора в профессиональный конвейер разработки

- **Platform:** Habr
- **Target length:** ~2500 words
- **Tags:** claude, ai, devtools, opensource, methodology, claude-code, automation
- **Hook:** Claude Code умеет всё. Проблема в том, что без инструкций он делает всё одновременно, непоследовательно и без тестов. Я написал плагин, который превращает его в разработчика с чётким пайплайном — от идеи до задеплоенного продукта.
- **CTA:** Плагин MIT-лицензирован: https://github.com/HiH-DimaN/idea-to-deploy

## Проблема: Claude Code без методологии — это хаос

Если вы пользовались Claude Code (CLI или IDE-расширение от Anthropic), вы знаете ощущение: говоришь «сделай SaaS для записи к парикмахеру» — и получаешь... что-то. Иногда хорошее, иногда нет. Но всегда без системы.

Типичные проблемы:
- **Код без архитектуры.** Claude сразу пишет код, не задумавшись о структуре БД, API-контрактах или деплое
- **Нет тестов.** Если не попросить явно — тестов не будет. А если попросить, они будут для последнего файла, а не для всего проекта
- **Потеря контекста.** Закрыл сессию — всё забыто. В следующей сессии Claude не знает, что было сделано, какие решения приняты, что дальше по плану
- **Нет ревью.** Код уходит в production без проверки. Claude не проверяет сам себя, если не заставить
- **Ad-hoc промптинг.** Каждый раз пишешь промпт заново. Никакого повторяемого процесса

Я прошёл через это на реальных проектах: SaaS-платформа с 5 LLM-провайдерами, AI-генератор бизнес-документов, мультиканальная платформа для LATAM-рынка. И на каждом сталкивался с одним и тем же — Claude Code мощный инструмент без инструкции.

## Решение: idea-to-deploy

[idea-to-deploy](https://github.com/HiH-DimaN/idea-to-deploy) — это плагин для Claude Code, который превращает хаотичную генерацию в структурированный пайплайн:

```
Идея → Вопросы → План → Архитектура → Код → Тесты → Ревью → Деплой
```

Установка одной командой:
```bash
/install github:HiH-DimaN/idea-to-deploy
```

После этого Claude Code получает **23 скилла**, **7 специализированных субагентов** и **12 enforcement-хуков**.

## Как это работает на практике

### Сценарий 1: Новый проект с нуля

Говоришь: «Создай SaaS-сервис для управления задачами фрилансеров, Telegram Mini App + FastAPI + PostgreSQL».

Что происходит:
1. **`/project`** — роутер определяет, что это новый проект, и направляет в `/kickstart`
2. **Уточняющие вопросы** — Claude спрашивает про авторизацию, монетизацию, деплой. Не принимает размытые ответы — переспрашивает с примерами
3. **Документация** — генерируется 6 документов: стратегический план, PRD с user stories, архитектура (БД-схема, API-эндпоинты, Docker), план реализации по шагам
4. **Ревью плана** — автоматически запускается `/review`, проверяет консистентность документов. Если что-то не сходится — фиксит до вашего одобрения
5. **Scaffolding** — создаёт структуру проекта: конфиги, Docker, .env.example, базовые модели
6. **Реализация по шагам** — каждый шаг из плана реализуется последовательно. После каждого шага — автоматические тесты
7. **Деплой** — Docker Compose, Nginx, health check

На выходе: работающий задеплоенный MVP с тестами, документацией и чистым git-логом.

### Сценарий 2: Ежедневная работа в существующем проекте

Говоришь: «У меня баг в auth».

Что происходит:
1. **Хук распознаёт** — триггер-фраза «баг» активирует подсказку: «используй `/bugfix`»
2. **`/bugfix`** — систематическая отладка: стек-трейс, git blame, бисект, fix, регрессионный тест
3. **Автоматический ревью** — после фикса запускается проверка

Или: «Отрефактори модуль авторизации» → `/refactor` по каталогу Фаулера. «Проверь безопасность» → `/security-audit` в стиле OWASP. «Подготовь к продакшену» → `/harden` с health checks, rate limiting, SRE runbook.

Один скилл `/task` — роутер для 15 типов задач. Описываешь что нужно — он направляет в правильный скилл.

## Что отличает от конкурентов

На рынке Claude Code плагинов есть альтернативы:

| | idea-to-deploy | gstack (70K stars) | BMAD (44K stars) | claude-code-skills |
|---|---|---|---|---|
| Скиллов | 23 | ~20 ролей | ~15 | 136 |
| Тестирование | 3-tier (CI + snapshot + headless) | Нет | Нет | Нет |
| Self-review | 25+ автоматических gates | Нет | Нет | Нет |
| Operations | migrate, harden, infra, migrate-prod | Нет | Нет | Частично |
| Enforcement | Блокирует после 3 игнорирований | Нет | Нет | Нет |
| Session persistence | Контекст сохраняется между сессиями | Нет | Нет | Нет |

**Ключевое отличие — не количество скиллов, а их качество.** У claude-code-skills более сотни скиллов — плоская коллекция без тестирования, без lifecycle, без self-review. Наши 23 проверяются 25+ автоматическими gates, каждый из которых появился из реального бага.

## Enforcement: Claude не может обойти методологию

Самый интересный аспект. В реальной 10-часовой сессии Claude **полностью проигнорировал** хуки-напоминания. Работал через raw Bash/Edit, не вызвав ни одного скилла. Напоминания каждые 60 секунд — и ноль реакции.

Почему? Потому что хуки были advisory-only. Напоминание без последствий = шум.

Решение (v1.19.0): **enforcement mode**. Хук считает игнорирования. После 3 подряд — **блокирует** следующий инструмент. Чтобы продолжить, Claude должен либо вызвать скилл, либо обосновать обход.

Это работает. Проверено live.

## Session persistence: контекст не теряется

Другая реальная проблема: закрыл сессию — потерял контекст. В следующей сессии Claude не знает, что было сделано, какие решения приняты.

Скилл `/session-save` автоматически сохраняет:
- Что сделано (конкретно, не "работал над проектом")
- Ключевые решения с обоснованием
- Блокеры
- Следующие шаги (actionable — можно начать выполнять сразу)
- Неочевидный контекст (устные договорённости, ограничения не из кода)

При старте следующей сессии хук `session-open-diagnostic` читает последний session-файл и показывает Claude полную картину: что было, что дальше по плану, какой ROADMAP актуален.

## Для кого это

- **Фрилансеры** — берёшь больше заказов, каждый с профессиональной архитектурой и тестами
- **Стартапы** — MVP за дни вместо недель. Один человек, но с пайплайном команды
- **Начинающие** — не нужно знать как структурировать проект, система делает это за тебя
- **Агентства** — стандартизация: каждый проект по одному процессу, предсказуемое качество

## Что дальше

Методология развивается с каждым реальным использованием. Каждый баг, каждое пользовательское наблюдение порождает новый автоматический gate. Система становится строже с каждым релизом — и это by design.
