## Когда методология ловит свои собственные баги

Мы написали плагин для Claude Code с 20 скиллами. Потом написали мета-ревью, которое проверяет сам плагин. А потом обнаружили, что каждый пользовательский баг-репорт порождает новый gate -- и система становится строже с каждым релизом.

### Контекст: что такое idea-to-deploy

idea-to-deploy -- это плагин для Claude Code (CLI/IDE расширение от Anthropic), который превращает хаотичную генерацию кода в структурированный пайплайн:

```
Идея -> Вопросы -> План -> Архитектура -> Код -> Тесты -> Ревью -> Деплой
```

20 скиллов покрывают весь жизненный цикл: от product discovery (MoSCoW/RICE приоритизация) до production hardening (health checks, rate limiting, graceful shutdown). 7 субагентов -- специализированные роли (архитектор, код-ревьюер, тест-генератор, анализатор производительности, документатор, бизнес-аналитик).

Проблема: 20 скиллов -- это 19 markdown-файлов с YAML-фронтматтером, ссылками друг на друга, триггер-фразами в хуках, строками в таблицах README, бейджами, версиями в plugin.json и marketplace.json. Любое изменение в одном месте может сломать консистентность в десяти других.

### Первая линия обороны: meta_review.py

`tests/meta_review.py` -- Python-скрипт, который запускается на каждом PR через GitHub Actions. Он проверяет ~25 структурных инвариантов и блокирует PR при любом Critical-уровне нарушении.

Примеры проверок:

**M-C1..M-C3: Версионная консистентность.** Версия в `plugin.json` должна совпадать с бейджами в `README.md`, `README.ru.md` и `marketplace.json`. Звучит тривиально, но при ручном обновлении забыть одно из четырёх мест -- нормальная ситуация.

```python
plugin_version = json.load(open("plugin.json"))["version"]
badge_match = re.search(r'Version-(.+?)-purple', readme_text)
assert plugin_version == badge_match.group(1), "Badge drift"
```

**M-C7: Количество скиллов.** Бейдж `Skills-19-green` должен совпадать с `len([d for d in skills/ if d.is_dir() and not d.name.startswith('_')])`.

**M-I7: Триггер-фразы.** Каждый скилл должен иметь хотя бы одну русскую и одну английскую триггер-фразу в хуке `check-skills.sh`. Если добавили новый скилл, но забыли прописать триггер -- хук не будет его предлагать пользователю.

Всё это работает. Но вот в чём подвох: структурные проверки ловят только те классы дрифта, о которых мы заранее подумали. А дрифт изобретателен.

### Цикл 1: marketplace.json (v1.13.2)

**Что произошло:** После полного аудита Anthropic compliance пользователь заметил, что в `marketplace.json` написано "seventeen skills + five subagents" (устаревшее значение), хотя реальное количество было другим. Но проблема была в другом: marketplace.json был создан вручную и с тех пор ни разу автоматически не проверялся.

**Почему существующие gates не поймали:** M-C7 проверяет бейдж в README. M-C1 проверяет version в plugin.json. Но marketplace.json -- отдельный файл со своей структурой, и ни один gate его не трогал.

**Что мы сделали:**
- **M-C13:** `marketplace.json plugins[0].version` == `plugin.json version`
- **M-C14:** Описание в marketplace.json должно содержать актуальные числа скиллов и субагентов

Две проверки, ~80 строк Python. Этот класс дрифта -- рассинхрон marketplace-файла с основным конфигом -- теперь покрыт навсегда.

### Цикл 2: Количество хуков в тексте README (v1.16.2)

**Что произошло:** Пользователь сказал: "в README секция хуков перечисляет 3 хука, а в директории hooks/ их 5". 

**Расследование:** Действительно, секция "Hooks" в README была написана при 3 хуках (pre-push, pre-commit, check-skills) и не обновлялась, когда добавили careful.sh и freeze.sh.

**Почему существующие gates не поймали:**
- M-C7 считает скиллы, не хуки
- M-C12 проверяет числа в прозе README, но с исключениями для заголовков (чтобы не ловить false positives на категорийных подзаголовках). Хуки упоминались в обычном тексте, но проверка была слишком груба-зернистой

**Что мы сделали:**
- **M-C15:** Количество хуков, упомянутых в нарративной секции README, должно совпадать с `len(hooks/*.sh)`

Одна проверка, ~60 строк. Класс "hook count в прозе" закрыт.

### Цикл 3: Подзаголовки категорий и полнота таблиц (v1.16.3)

**Что произошло:** Пользователь посчитал скиллы в скобках категорийных заголовков README:

```markdown
### Entry Points (2 skills)
### Project Creation (3 skills)
### Daily Work (6 skills)
### Operations (4 skills)     <-- но в таблице 3 строки!
### Session Management (1 skill)
```

Сумма: 2+3+6+**4**+1 = 19. Но реальное число: 18. "Operations (4 skills)" -- дрифт, который существовал давно и пережил 22 PR.

Потом тот же пользователь сказал: "а в таблице Skill Contracts я вижу 17 строк, не 18". Расследование показало: `/task` (добавлен в v1.5.0) был в таблице Entry Points, но отсутствовал в четырёх comprehensive-таблицах (Skill Contracts, Recommended Models и их русских аналогах). 11 месяцев и 22 PR -- и никто не заметил.

**Почему существующие gates не поймали:**
- M-C7 проверяет бейдж (Skills-18 == 18 директорий) -- проходит
- M-C12 пропускает строки с заголовками (`if heading_line_re.match(line): continue`) -- by design, чтобы не фолсить на подзаголовках
- M-I4 проверяет "скилл упомянут где-нибудь в README" -- проходит, потому что `/task` есть в Entry Points

**Что мы сделали:**
- **M-C16, Mode A:** Парсит `### Category (N skills)`, считает строки таблицы после заголовка, проверяет N == row_count. Дополнительно считает сумму всех подзаголовков и сравнивает с `len(skills/)`.
- **M-C16, Mode B:** Для каждой comprehensive-таблицы (4 маркера: "Skill Contracts", "Recommended Models", "Контракты скиллов", "Рекомендуемые модели") извлекает все `/skill-name` упоминания и проверяет равенство множеств с директориями skills/.

~140 строк Python. Два режима, потому что это два разных класса дрифта, обнаруженных одним пользователем.

### Паттерн

Посмотрим на это сверху:

| Цикл | Наблюдение пользователя | Класс дрифта | Новые gates |
|---|---|---|---|
| v1.13.2 | "marketplace.json не синхронизирован" | marketplace vs plugin.json | M-C13, M-C14 |
| v1.16.2 | "в README не все хуки" | hook count в прозе | M-C15 |
| v1.16.3 | "в скобках 19, а скиллов 18" + "в таблице 17" | категорийные подзаголовки + полнота таблиц | M-C16 (2 режима) |

4 наблюдения -> 5 новых gates за 8 дней.

Каждый gate стоит ~50-150 строк Python. Каждый навсегда закрывает класс дрифта. Маржинальная стоимость низкая, маржинальная польза высокая.

Что критично: ни одно из этих наблюдений не было "багом" в обычном смысле. Плагин работал. Скиллы генерировали правильный output. Но метаданные дрифтовали -- и это подрывало доверие пользователей и создавало путаницу.

### Три тира тестирования

Self-improvement loop работает на первом тире. Но у нас их три:

**Тир 1: Структурный (CI, бесплатно, 5 секунд).** `meta_review.py` с 25+ проверками. Запускается на каждом PR. Блокирует мерж при Critical-нарушении. Сюда попадают все новые gates из loop'а.

**Тир 2: Snapshot (локально, бесплатно, <1 секунды).** `verify_snapshot.py` валидирует генерированный output по контракту `expected-snapshot.json`. Проверяет: наличие файлов, секции в документах, маркеры контента (чтобы LLM не генерил generic текст, игнорируя домен пользователя), числовые ограничения (минимум API эндпоинтов, минимум user stories).

```json
{
  "content_contracts": {
    "PROJECT_ARCHITECTURE.md": {
      "required_sections": ["Competitors|Конкуренты"],
      "min_api_endpoints": 15,
      "must_contain_any_of": {
        "competitors_named": ["MEDODS", "IDENT"]
      }
    }
  }
}
```

**Тир 3: Поведенческий (headless Claude Code, $2-8 за прогон, 10-20 минут).** `run-fixture-headless.sh` запускает `claude -p` в non-interactive режиме, скармливает JSONL с идеей и заранее подготовленными ответами на уточняющие вопросы, ждёт генерации, запускает verify_snapshot.py на результате.

```bash
claude -p \
  --input-format stream-json \
  --output-format stream-json \
  --verbose \
  --model sonnet \
  --dangerously-skip-permissions \
  --max-budget-usd 10.00 \
  < fixture.jsonl
```

Три фикстуры активны, семь в статусе pending. Стоимость полного прогона: $20-80. Запускаем только на release-ветках.

### Почему это работает

**Распределённый аудит.** Человеческое pattern matching ловит дрифт, который автоматические структурные gates пропускают. Каждое наблюдение кодируется как gate, и одно и то же наблюдение больше не нужно делать вручную.

**Низкий порог входа.** Добавить gate -- это написать одну Python-функцию с assert'ами и зарегистрировать её в списке проверок. Не нужно менять CI pipeline, не нужно менять инфраструктуру.

**Инкрементальность.** Каждый gate ловит ровно один класс дрифта. Не пытаемся написать "универсальную проверку". Вместо этого выращиваем набор специфичных проверок, каждая из которых появилась из реального бага.

**Эмпирическое подтверждение.** Каждый новый gate валидируется в обе стороны: запускаем его на сломанной версии (должен fail) и на исправленной (должен pass). Это не теоретический дизайн -- это зафиксированные случаи.

### Конкурентный контекст

Конкуренты в пространстве Claude Code плагинов:

- **gstack** (70K stars) -- набор ролей (role-based), нет self-review
- **BMAD** (44K stars) -- agile-фреймворк, нет мета-ревью, нет operations-скиллов
- **claude-code-skills** (136 скиллов) -- плоская коллекция, без тестирования, без lifecycle

Наша дифференциация -- не количество скиллов (у claude-code-skills их 136 против наших 19). А то, что наши 19 проверяются 25+ автоматическими gates, каждый из которых появился из реального бага. Плюс operations-стек (migrate/harden/infra), daily-work роутер, и трёхуровневое тестирование.

### Что дальше

**v1.18+:** Каждое следующее пользовательское наблюдение, которое найдёт непокрытый класс дрифта, породит новый gate. Система становится строже с каждым циклом.

**Pending fixtures:** 7 фикстур ждут активации. Каждая -- это один headless прогон для калибровки snapshot-контракта.

**Стоимость CI:** При 4 релизах в месяц полный поведенческий прогон стоит $80-320/месяц. На Claude Pro подписке фактическая стоимость $0 за API, только подписка.
