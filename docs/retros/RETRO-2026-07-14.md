# RETRO — session hygiene exercises (2026-07-14)

## Scope

Практическая проверка изменений session hygiene/quality lifecycle из ветки
`codex/host-adapter-session-hygiene`:

1. exit checklist на пяти последовательных сессиях;
2. A/B-бенчмарк одного фиксированного набора задач со strict-clean harness и
   verification-only вариантом;
3. reversible ablation `completion-stop` с решением оставить/удалить/заменить.

Эксперименты используют настоящие `docs/templates/itd/itd_hygiene.py` и
`hooks/completion-stop.sh` в изолированных временных Git-репозиториях. Дефекты
посеяны намеренно и детерминированно. Поэтому результаты доказывают способность
гейта различать заданные классы состояний, но не оценивают частоту таких
дефектов в production-сессиях.

## Упражнение 1 — пять последовательных сессий

Exit checklist проверял пять независимых измерений:

| Измерение | Исполняемое доказательство |
|---|---|
| Verification | Команда build/test из `VERIFICATION_CONTRACT.json` возвращает 0 |
| Durable state | Каждый `requiredStateFiles` существует |
| Debug hygiene | В заданном scope нет debug-маркеров |
| Startup | Стандартная startup/smoke-команда возвращает 0 |
| Cleanliness | Manifest cleanup выполнен, затем `git status` пуст |

Пять прогонов выполнялись последовательно в одной временной кодовой базе.
После каждого красного close нарушение исправлялось и выполнялась одна повторная
попытка.

| Сессия | Посеянное состояние | Нарушенные измерения | First pass | Ретраи | Final |
|---|---|---|---:|---:|---:|
| S1 | чистое | — | PASS | 0 | PASS |
| S2 | verification exit 1 | verification | FAIL | 1 | PASS |
| S3 | отсутствует state-файл | durable state | FAIL | 1 | PASS |
| S4 | debug marker + untracked scratch | debug hygiene, cleanliness | FAIL | 1 | PASS |
| S5 | startup exit 1 | startup | FAIL | 1 | PASS |

Итог:

- каждое из 5 измерений получило ровно одно контролируемое нарушение;
- гейт обнаружил все `5/5` классов, включая два нарушения за один прогон S4;
- first-pass completion: `1/5 = 20%`;
- после remediation: `5/5 = 100%`;
- ретраи: `4`;
- false clean exits: `0`.

## Упражнение 2 — strict-clean против verification-only

Оба варианта получили один фиксированный набор из 10 задач:

| Task | Посеянные дефекты |
|---|---|
| T01 | clean |
| T02 | verification failure |
| T03 | missing state |
| T04 | debug marker |
| T05 | startup failure |
| T06 | dirty worktree |
| T07 | debug marker + dirty worktree |
| T08 | missing state + startup failure |
| T09 | manifest-owned temporary artifact |
| T10 | clean repeat |

`strict-clean` запускал полный explicit-close contract. `verification-only`
проверял ту же build/test-команду, но не требовал state/debug/startup/cleanup/
clean-Git. После явного FAIL разрешалась одна remediation-попытка.

| Метрика | Strict clean | Verification only | Δ strict |
|---|---:|---:|---:|
| First-pass completed | 3/10 (30%) | 9/10 (90%) | −60 п.п. |
| Final completed | 10/10 (100%) | 10/10 (100%) | 0 п.п. |
| Ретраи | 7 | 1 | +6 |
| Completed tasks с leaked defect | 0/10 (0%) | 7/10 (70%) | **−70 п.п.** |
| Leaked defect instances | 0 | 9 | −9 |

T09 подтвердил немедленную cleanup-ветку: session-owned artifact был удалён
до clean-Git проверки и задача прошла с первой попытки. Активный weekly-runner
дополнительно выполнен два раза подряд; оба прогона завершились тихо с exit 0,
то есть повторный tracing cleanup не создал дрейф.

Интерпретация: final completion не ухудшился, но strict harness сделал
невидимую работу явной. Цена на этом специально дефектном наборе — шесть
дополнительных ретраев; выигрыш — отсутствие просочившихся дефектов вместо 70%.

## Упражнение 3 — ablation completion-stop

Компонент: `hooks/completion-stop.sh`.

Kill switch: `ITD_COMPLETION_STOP=0`.

Фиксированный suite содержит шесть случаев: грязный Python, JavaScript и SQL
(ожидается предупреждение), docs-only dirty, clean source и recursive Stop
(предупреждение не ожидается). Monthly runner выполнил один suite сначала в
baseline, затем с kill switch.

| Вариант | Правильная классификация | Пропущенные предупреждения |
|---|---:|---:|
| Enabled | 6/6 (100%) | 0 |
| Disabled | 3/6 (50%) | 3: Python, JavaScript, SQL |

Машинный результат ablation:

```json
{
  "component": "completion-stop-suite",
  "baselineMetric": 6.0,
  "disabledMetric": 3.0,
  "regressed": true,
  "recommendation": "restore",
  "humanDecision": "pending"
}
```

Решение этого упражнения: **KEEP**. Отключение убирает 50% полезной
классификации suite и теряет все три целевых предупреждения. Замена более лёгким
компонентом не обоснована: измерения стоимости/latency не показали проблемы,
а функциональная регрессия уже подтверждена.

## Повторная оценка пяти изменений

| Пункт | Было после реализации | После практики | Evidence / оставшийся вычет |
|---|---:|---:|---|
| Clean state как exit requirement | 5.0 | **5.0** | 5/5 классов нарушений пойманы; false clean exits 0 |
| Немедленная + периодическая очистка | 4.5 | **4.7** | T09 + два weekly exit 0; реальный scheduler всё ещё внешний |
| Активный quality document | 4.5 | **4.5** | Валидируется каждым strict/weekly run; содержательная оценка модулей human/evidence-reviewed |
| Периодическое упрощение harness | 4.5 | **4.8** | Monthly ablation реально нашла 6→3 и fail-closed предложила restore; календарный запуск внешний |
| Идемпотентные cleanup operations | 5.0 | **5.0** | Повтор weekly зелёный; manifest-owned missing artifact остаётся no-op |

Средняя практическая оценка: **4.8/5** (до реализации 2.8; после статической
проверки 4.7; после воспроизводимых упражнений 4.8).

## PROPOSALS

Новых release-кандидатов по этим синтетическим данным нет. Смягчать clean-state
gate ради уменьшения ретраев было бы Goodhart-ошибкой: final completion одинаков
(100%), а leakage без гейта вырос до 70%.

Единственная операционная заметка — подключить weekly/monthly команды к реальному
scheduler, когда появится внешний сигнал пропущенной периодичности. Один
контролируемый прогон не является достаточным production-сигналом для отдельного
релиза.
