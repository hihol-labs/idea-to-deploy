# Project-level проверки — `.itd/checks/` (v1.87.0)

First-class слот для исполняемых архитектурных правил ПРОЕКТА (не методологии).
Происхождение: retro 2026-07-11, сет-3 упражнений harness engineering, упр. 2 —
правила («BigInt-сериализация только через replacer», «номер миграции не занят
в main») конвертируются в проверки, но штатного места для них не было.

## Контракт

- Проверки — файлы в `<project>/.itd/checks/`; диспатч по расширению:
  `*.sh` → sh, `*.mjs|*.js` → node, `*.py` → `itd_py.sh` (обход WindowsApps-шима).
- Раннер: `skills/_shared/itd_project_checks.sh [--root DIR] [--list]
  [--files f1 f2 …]`. Изменённые файлы приходят проверке в env
  `ITD_CHANGED_FILES` (по одному на строку) — скоуп opt-in, whole-repo валиден.
- Exit 0 = чисто; любой ненулевой exit проверки → раннер exit 1 с её выводом.
- **Сообщения проверок обязаны быть agent-oriented**: WHY (что сломается и
  когда) + FIX (конкретная команда/паттерн с образцом) + file:line. Образцы:
  проверки OneOfS (`check-doc-json-serialization.mjs`, `check-migration-numbers.sh`).

## Интеграция в жизненный цикл

- **DoD-прекоммит** (`hooks/check-dod-before-commit.sh`): при наличии
  `.itd/checks/` раннер гоняется на staged-файлах; красная проверка →
  deny коммита. Обход — штатный `SKILL_BYPASS: <причина>` в description
  Bash-вызова коммита (пишется в bypass-леджер).
- Вручную/из скиллов: `sh "$SHD/itd_project_checks.sh" --root .`.
- Best-effort invariant: нет каталога / раннера / node — no-op или явное
  «НЕ ПРОВЕРЕНА», никогда не ложный green.

## Гигиена

- `.itd/` в целевых проектах обычно вне git (`.git/info/exclude`) — проверки
  локальны для машины; шарить с напарником — через отдельное решение владельца
  репо (проверка может жить и в репо проекта, если команда согласна).
- Проверка должна быть быстрой (< пары секунд): она стоит на пути каждого
  коммита. Тяжёлое — выноси в CI проекта, не в слот.
